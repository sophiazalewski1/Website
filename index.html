Parallel Graph Algorithm Implementations in GraphLab and C++

We present various parallel graph algorithm implementations to explore the trade-offs between domain specific and lower level programming languages.

Summary

This project implements various parallel graph algorithms to explore the trade-offs between using a higher level, domain specific programming language (such as GraphLab) and a general lower level language (C++ with OpenMP). We compare the implementations of:
  • Ford Fulkerson's Algorithm for Max-Flow,
  • Dinic's algorithm for Max-Flow
  • Page Rank for determining web search results (if time).

Together these algorithms require a wide range of parallel tasks including graph search, contraction, and clustering. We then evaluate the effectiveness of the parallel frameworks using two main metrics: ease of programming and performance.

Background

1. Max Flow

Max Flow is a classic graph optimization problem where the goal is to push some abstract ‚Äúflow" across edges in a directed graph from some source vertex ùë† to some sink ùë°. More concretely, let ùê∫ = (ùëâ , ùê∏) be a directed graph. Each (ùë¢, ùë£) ‚àà ùê∏ has some positive capacity ùëê (ùë¢, ùë£) indicating how much flow can be pushed from ùë¢ to ùë£. The goal is to find some flow ùëì which can be thought of as a function mapping edges to how much flow is being pushed through them. This flow must have the following properties:

$$0 \leq f(u,v) \leq c(u,v) \\ \sum_u f(u,v) = \sum_u f(u,v) \text{ for all } v \notin \{s,t\}$$

where the first constraint enforces that no edge has more flow than its capacity and the second condition enforces that for all vertices other than ùë† and ùë°. The simplest algorithm for finding max flow is called Ford Fulkerson. Ford Fulkerson works by repeatedly finding a path from ùë† ‚àí ùë° that increase the overall flow from ùë† ‚àí ùë° and updating some state. The ‚Äústate" is called the residual graph ùê∫‚Ä≤ and allows the algorithm to backtrack if necessary. Ford Fulkerson finds ‚Äúaugmenting paths" by running a graph search with source ùë† and target ùë° on the residual graph. Here, we can use a parallel implementation of Breadth First Search (BFS).

A more complicated yet efficient algorithm, Dinics, finds a set of augmenting paths called a ‚Äúblocking flow" each iteration by making up to |ùê∏| calls to depth first search. In Dinics it is therefore possible to add an addition axis of parallelism in finding blocking flows. Another extension to the Max Flow problem introduces a cost function to each edge that charges a flow $(ùë¢, ùë£) for each unit of flow pushed through (ùë¢, ùë£). The objective becomes finding a maximum flow that also minimizes cost. If we have additional time, we will explore these extensions to the Ford Fulkerson algorithm [2].

3. PageRank (if time)

PageRank is a newer graph algorithm designed by Larry Page and Sergey Brin for ranking the importance of web pages for their search engine, Google. The algorithm judges the importance of a website by estimating the probability that someone randomly clicking links will end up on that page. By representing the hyperlinks as a graph where an edge connects page ùê¥ to page ùêµ if ùê¥ contains a link to ùêµ, the probability distribution can be calculated iteratively. Each iteration, the PageRank value of a vertex is distributed evenly among its outgoing neighbors while at the same time its new PageRank value is computed by combining the contributions from its incoming neighbors. This process repeats until the graph converges [4]. Because PageRank is defined in terms of an update policy for each vertex at every iteration, it is relatively easy to program in GraphLab. In a lower level framework like C++ and OpenMP there are a lot more ways to organize the computations and the schedule for the updates. One of the goals of our project will be to see if we can beat the performance of the GraphLab implementation using this greater flexibility.

Challenges

  • Graph algorithms contain dependcies between any paths that share an edge
  • Very little locality in graph traversal algorithms, each edge in a path is in a different cache-line
  • Low computation to communication ratio
  • Divergent execution since no two runs of a graph traversal traverse the same path
  • CMU's Graphlab is deprecated meaning that we need to design our own GraphLab-like language, which we will call GraphLabLite, to test our implementations on this interface

Resources

For this project, we are using Ford Fulkerson starter code from programiz and Dinic's starter code from geeksforgeeks, which we will use as our correctness checks for our parallel algorithms. After implementing and parallelizing our algorithms, we will be running our implementations on the Gates and PSC machines to test our code on a high number of CPUs and measure performance and scalability. Additionally, we will also design our own Graph programming lanuage (GraphLabLite) and measure the performance of the algorithms in this Graph interface vs. standard OpenMP.

Goals and Deliverables

1. Plan to Achieve

We plan to be able to implement at least 2 of the 3 graph algorithms described above in OpenMP. Additionally, we also hope to implement our own GraphLab-like interface, GraphLabLite, and create versions of each algorithm that can be represetned in this interface. We aim to show numerous graphs comparing each of our implementations (Ford Fulkerson and Dinics using our various BFS parallziations as well as using our GraphLabLite implementation) across graphs of varying types and sizes. For example, we may choose to show the timing of each implementation on dense layer graphs of varying number of nodes and varying number of layers, or compare the implementation on graphs with the same num- ber of nodes but different ‚Äùtypes‚Äù / other properties (hence the importance our flexible graph generating script). Moreover, we will also produce plots of our results scaling with the number of processors.

2. Hope to Achieve

If time allows, we will also implement the PageRank algorithm to have a further metric of comparison between GraphLabLite, which we aim to implement, and OpenMP. Furthermore, if we are extra ahead of schedule we will also consider an implementation in MPI, since for very large graphs message passing might be more efficient than OpenMP, given that very large graphs may be too large to fit in one memory system. If we are struggling to complete all of the implementations on all of the platforms, we will simply focus on one algorithm and do a more in-depth analysis on the differences in OpenMP and GraphLabLite for that specific algorithm.

Platform Choice

We are exploring our own form of GraphLab for one of our implementation platforms because this language is specifically designed and optimized for parallelizing graph algorithms. We will compare this to OpenMP, since this a shared memory platform, and graph problems will require updating lots of shared memory locations. We are mainly considering CPU implementations rather than GPU, because graph traversal algorithms are highly divergent and may not benefit from GPUs.
